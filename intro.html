Getting Started

Last updated: June 13th, 2022

Intro


In this introductory page, we briefly talk about Znuny’s design system and the logic behind its components, as well as how to build them.

The design system for the Znuny platform is made with the principles of Atomic Design in mind (originally developed by Brad Frost), wherein every component you see is made up of atoms. These atoms are combined to create molecules, which are themselves combined to create larger components called organisms. Finally, these organisms are combined to create complete user interfaces. The idea behind atomic design is that by breaking interfaces down into their smallest components, designers can create more flexible, scalable, and efficient design systems. This allows them to more easily create consistent designs across multiple platforms and devices, and to quickly iterate on designs as needed.

In this way, we started creating a library of reusable (and editable) components that are both easy and intuitive to use. However, it is important to note that we're building on top of an already existing system, so we were met with a couple of limitations. Some things had to be left mostly untouched, as far as HTML structuring goes, as any change in that would require major back-end restructures, which would have been extremely time-consuming in the end.

The Znuny platform operates on a system of Widgets. Every window-like feature you see is probably a Widget.

Most every Widget contains a Header and a Content div. There are some exceptions where Widgets are Widgets, but they don’t look much like one (this can be seen in the ticket table in the queue view, for example). We’ve preserved some of these exception Widgets, for the time being.

Some CSS considerations

CSS rules are using the flexbox model due to its adaptability, which makes responsive interfaces a lot easier to achieve. Ideally, this is the styling model that should be followed at all times.

    CSS files have also been reorganized, separated into 3 categories:
  • - the base styles (generic page and form setups, for example);
  • - the page-specific styles (legacy);
  • - the components style (new addition);

The page-specific files specify styles that are unique to certain pages within the platform, like Agent.Preferences.css. This type of logic currently goes against the component-only principle and should be generally avoided.
